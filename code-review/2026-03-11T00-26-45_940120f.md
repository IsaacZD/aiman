## Overall Impression

This is a well-structured, clean codebase. The Rust agent is idiomatic, async-first, and properly
layered. The TypeScript dashboard is readable and pragmatic. The main areas worth attention are
correctness edge cases, silent error handling, and some design gaps.

---

## Bugs / Correctness

**1. `stop()` has misleading behavior on naturally-exited engines** (`supervisor.rs:381`)

`stop_tx` is set when the engine starts and never cleared after natural exit. Calling `stop` on a
naturally-exited engine sends to the watch channel and returns `Ok` with `Stopped` status ŌĆö but
`NotRunning` should be returned. The check should also test `task.is_finished()`:

    // Current: only checks stop_tx is Some
    // Missing: task.is_finished() check means stop on a dead engine returns Ok

**2. `persist_config_store` and `append_jsonl` silently swallow errors** (`supervisor.rs:280, 639`)

Both use `let _ = ...write(...)` with no logging. If a write fails (disk full, permissions),
in-memory state diverges from disk with no indication.

    // Should at minimum: tracing::error!("failed to persist config store: {err}");

**3. `retries` counter never resets in `run_engine`** (`supervisor.rs:420`)

If an engine runs for hours, crashes, restarts, and then crashes again later ŌĆö all crashes count
toward the same `max_retries` budget. Most process supervisors reset the counter if the process
was healthy for some minimum duration (e.g., 30s). Right now `max_retries = 5` could be exhausted
across crashes days apart.

**4. Benchmark ID collision** (`benchmark.rs:143`)

`format!("bench-{}", Utc::now().timestamp_millis())` is not unique under concurrent runs. Use a
UUID (`uuid` crate, feature `v4`) or at least append random bytes.

**5. `read_jsonl` double-parses each line** (`supervisor.rs:663-688`)

Each line is deserialized to `T`, then `extract_ts` re-deserializes to `serde_json::Value` to
extract the timestamp. If `T` has a `ts: String` field, the timestamp is already available after
the first parse.

## Security

**6. Timing attack on token comparison** (`auth.rs:26`)

    if provided == Some(expected.as_str()) { ... }

String equality is not constant-time. For a LAN-only tool this is low risk, but
`subtle::ConstantTimeEq` or `hmac::equal` would be trivially correct.

**7. No timeout on proxied HTTP in the dashboard** (`server/index.ts`)

Most proxy endpoints (hardware, configs, start, stop) use `fetch` with no timeout. An unresponsive
agent will hang the dashboard request indefinitely. The dashboard benchmark does use
`AbortController` correctly ŌĆö apply the same pattern to all proxy calls.

**8. `loadHosts()` reads disk on every request** (`server/index.ts:132-462`)

Every handler calls `loadHosts()` which reads and parses JSON from disk. For <5 hosts this is fine
functionally, but a stuck filesystem or slow disk means all API requests stall. A simple in-memory
cache with dirty-flag invalidation on writes would fix this.

## Design / Architecture

**9. Duplicated benchmark logic** (`benchmark.rs` vs `server/index.ts`)

The two benchmark implementations (Rust for agent-side, TypeScript for dashboard-side) are
near-identical:
- `generatePrompt`/`generate_prompt` share the same word pool
- `summarize_results`/`runBenchmarkConcurrency` are parallel implementations
- `inferApiBase`, `parseArgValue`, `defaultPort`, `percentile`, `countWords` ŌĆö all duplicated

When you add a new engine type or change default ports, you need to update both. Consider making
the dashboard benchmark always delegate to the agent rather than re-implementing it.

**10. `stop()` is fire-and-forget but returns the pre-stop status** (`supervisor.rs:222-228`, `api.rs:62-74`)

The stop API sends a signal and immediately returns the instance, which still has `Running` status.
The caller must poll to see `Stopped`. This is fine async design, but the HTTP response could
include a hint ŌĆö or the status could transition to `Stopping` between signal-sent and
process-exited.

**11. No `Stopping` status**

There's a window between `stop()` being called and the process actually dying where the status is
still `Running`. Adding a `Stopping` variant to `EngineStatus` would make the UI more accurate and
let you prevent double-stop.

**12. `EngineHandle::clone_for_task` manual field list** (`supervisor.rs:391-402`)

`EngineTaskHandle` is a manual subset clone of `EngineHandle`. If you add a field to `EngineHandle`
that should be shared with the task, it's easy to forget to add it to `clone_for_task`. Wrapping
the shared fields in an `Arc<EngineShared>` inner struct would eliminate this.

**13. `EngineInstance.id` and `config_id` are always identical** (`supervisor.rs:327-330`)

Both are set to `config.id`. If you never plan to run multiple instances per config, `config_id` is
redundant. If you do plan for it, document the intent.

**14. `EngineInstance.health` is always `None`** (`lib.rs:62`)

The field exists but is never set. Either populate it (e.g., from a health-check endpoint probe) or
remove it to avoid confusion.

## Minor Code Quality

**15. `&PathBuf` parameters should be `&Path`** (`supervisor.rs:244, 276, 629, 646`)

Rust convention is to accept `&Path` (or `impl AsRef<Path>`) rather than `&PathBuf` for function
parameters. This is more flexible and idiomatic.

**16. `scan_models` has an unnecessary clone** (`api.rs:301-302`)

    let libraries = request.libraries.clone();
    let artifacts = tokio::task::spawn_blocking(move || scan_model_libraries(&libraries))

`request.libraries` is owned; you can move it directly into the closure without cloning.

**17. WebSocket proxy doesn't handle upstream errors** (`server/index.ts:498-516`)

If the upstream agent WebSocket fails to connect, `upstream.on("error", ...)` is not handled.
In Node.js, unhandled error events crash the process. Add an `upstream.on("error", ...)` handler.

## Missing / Future Considerations

- **No automated tests**: First candidates ŌĆö unit tests for `validate_config`, `should_restart`,
  `percentile`, and `summarize_results` in Rust (all pure functions with no I/O).
- **No `PATCH /v1/configs/{id}`**: Updates require sending the full config. A partial update
  endpoint would be friendlier for the UI.
- **Log file growth**: JSONL files grow unboundedly. Consider a rotation policy (max size or max
  age), especially for noisy engines.
- **WebSocket auth from browsers**: The WebSocket upgrade goes through the auth middleware
  (checked at upgrade time), which is correct ŌĆö but browsers can't send custom headers on WS
  upgrades. A client connecting directly to the agent (not via the dashboard bridge) would need
  query-param token support.
